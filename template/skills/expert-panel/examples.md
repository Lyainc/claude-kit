## Example 1: API 설계 검토

### 사용자 요청

```
User: 이 REST API 설계 문서를 보안전문가, 성능전문가, UX전문가 관점에서 검토해줘.
      인증 방식과 페이지네이션 전략이 적절한지 확인하고 싶어.
```

### Phase 0: 토론 준비

```
[모더레이터]
토론을 준비하겠습니다.

**검토 대상**: REST API 설계 문서
**전문가 집단**: 보안전문가, 성능전문가, UX전문가 (3명)
**토픽 분할**:
  1. 인증 방식 (JWT vs Session)
  2. 페이지네이션 전략 (Offset vs Cursor)
  3. 에러 응답 형식

토론을 시작하겠습니다.
```

### Phase 1: Topic 1 - 인증 방식

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 TOPIC 1: 인증 방식
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎤 [긍정적 실무자]:
JWT + Refresh Token 방식을 제안합니다.
- Stateless하여 서버 확장에 유리
- 모바일/웹 클라이언트 모두 지원 용이
- Access Token 15분, Refresh Token 7일로 설정 권장

🎤 [부정적 실무자]:
JWT 방식의 리스크를 지적합니다.
- Token 탈취 시 만료까지 무효화 불가
- Payload 크기로 인한 네트워크 오버헤드
- Refresh Token 저장 위치 보안 이슈

🎤 [보안전문가]:
Refresh Token Rotation은 적용 예정인가요?
탈취 감지를 위한 추가 메커니즘이 필요해 보입니다.

🎤 [긍정적 실무자]:
네, Refresh Token Rotation과 함께
Device Fingerprint 검증을 추가할 수 있습니다.

🎤 [성능전문가]:
JWT 검증 시 매 요청마다 signature 검증이 필요한데,
Redis 캐싱으로 검증 결과를 저장하면 부하를 줄일 수 있습니다.

🎤 [UX전문가]:
사용자 관점에서 Silent Refresh가 잘 동작한다면
세션 방식과 UX 차이가 없을 것 같습니다.

🎤 [부정적 실무자]:
보안전문가의 우려가 타당합니다.
Refresh Token Rotation + Device Fingerprint 조건으로
JWT 방식에 동의할 수 있습니다.

✅ CONCLUSION:
JWT + Refresh Token 방식 합의
조건: Refresh Token Rotation, Device Fingerprint 검증 필수
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Phase 1: Topic 2 - 페이지네이션

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 TOPIC 2: 페이지네이션 전략
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎤 [긍정적 실무자]:
Cursor 기반 페이지네이션을 제안합니다.
- 실시간 데이터 변경에 일관성 유지
- 대용량 데이터에서 성능 우수 (O(1) vs O(n))

🎤 [부정적 실무자]:
Cursor 방식의 단점도 있습니다.
- 특정 페이지로 직접 이동 불가
- 구현 복잡도 증가
- 정렬 기준 변경 시 cursor 무효화

🎤 [UX전문가]:
관리자 페이지에서는 "5페이지로 이동" 기능이 필요한데,
Cursor 방식으로는 어떻게 처리하나요?

🎤 [긍정적 실무자]:
관리자용 API와 사용자용 API를 분리하여
관리자는 Offset, 사용자는 Cursor를 사용할 수 있습니다.

🎤 [성능전문가]:
하이브리드 방식은 유지보수 부담이 됩니다.
사용자 API만이라도 Cursor로 통일하는 게 좋겠습니다.

🎤 [부정적 실무자]:
사용자 API는 무한 스크롤이 대부분이므로 Cursor가 적합합니다.
하이브리드 제안에 동의합니다.

✅ CONCLUSION:
하이브리드 방식 합의
- 사용자 API: Cursor 기반
- 관리자 API: Offset 기반 (데이터량 제한적)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Phase 1: Topic 3 - 에러 처리 (보류)

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 TOPIC 3: 에러 응답 형식
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎤 [긍정적 실무자]:
RFC 7807 (Problem Details) 표준을 제안합니다.
상세한 에러 정보로 디버깅에 유리합니다.

🎤 [보안전문가]:
상세 에러 정보는 공격자에게 시스템 정보를 노출합니다.
프로덕션에서는 최소 정보만 반환해야 합니다.

🎤 [UX전문가]:
하지만 사용자에게 "문제가 발생했습니다"만 보여주면
무엇을 잘못했는지 알 수 없습니다.

🎤 [보안전문가]:
사용자 입력 오류와 시스템 오류를 구분해야 합니다.
4xx는 상세히, 5xx는 최소화하는 방식이 필요합니다.

🎤 [부정적 실무자]:
환경별로 다른 응답을 주는 것은 테스트/운영 불일치를 야기합니다.

[모더레이터]
의견이 교착 상태입니다.
에러 응답 정책에 대한 보안 가이드라인이 있는지 확인이 필요합니다.

**중단 사유**: 팩트체크 필요
**필요 정보**: 회사 보안 정책에서 API 에러 응답 관련 가이드라인
**재개 조건**: 정책 확인 후 재논의

⏸️ CONCLUSION: 보류 - 보안 정책 확인 필요
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## Example 2: 정책 문서 검토

### 사용자 요청

```
User: 재택근무 정책안을 법률전문가, HR전문가, 현장관리자 관점에서 검토해줘.
```

### Phase 0: 토론 준비

```
[모더레이터]
토론을 준비하겠습니다.

**검토 대상**: 재택근무 정책안
**전문가 집단**: 법률전문가, HR전문가, 현장관리자 (3명)
**토픽 분할**:
  1. 근태 관리 방식
  2. 업무 보안 규정
  3. 비용 처리 기준

토론을 시작하겠습니다.
```

### Phase 1: Topic 1 - 근태 관리

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 TOPIC 1: 근태 관리 방식
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎤 [긍정적 실무자]:
성과 기반 평가로 전환을 제안합니다.
- 출퇴근 시간보다 결과물 중심 평가
- 코어타임(10-16시)만 필수 참여
- 주 1회 진행 보고로 관리

🎤 [부정적 실무자]:
성과 기반만으로는 리스크가 있습니다.
- 협업이 필요한 업무에서 응답 지연 발생
- 신입사원 온보딩 어려움
- 업무량 불균형 시 형평성 문제

🎤 [법률전문가]:
근로기준법상 근로시간 기록은 필수입니다.
성과 기반이어도 시간 기록 시스템은 유지해야 합니다.

🎤 [HR전문가]:
직급별 차등 적용은 어떨까요?
- 시니어: 성과 기반 + 코어타임
- 주니어: 시간 기반 + 멘토링 시간 확보

🎤 [현장관리자]:
팀별로 상황이 다릅니다.
개발팀은 성과 기반이 맞지만, 운영팀은 교대 근무가 필요합니다.

🎤 [긍정적 실무자]:
직급별 + 직무별 차등 적용에 동의합니다.
다만 기본 원칙은 성과 기반으로 설정하고 예외를 두는 방식이 좋겠습니다.

🎤 [부정적 실무자]:
법률전문가 의견대로 시간 기록은 유지하되,
평가 기준은 성과 중심으로 하는 절충안에 동의합니다.

✅ CONCLUSION:
하이브리드 방식 합의
- 기본: 성과 기반 + 코어타임(10-16시)
- 시간 기록: 법적 요건 충족용 유지
- 예외: 운영팀 등 교대 필요 직무
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## Example 3: 모더레이터 개입 (재논의)

### 전문가 2명 이상 반대 시

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 TOPIC 2: 데이터 저장 위치
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🎤 [긍정적 실무자]:
AWS S3를 메인 스토리지로 제안합니다.

🎤 [보안전문가]:
개인정보가 포함된 데이터는 국내 리전이 필수입니다.
AWS 서울 리전만으로는 재해복구가 불충분합니다.

🎤 [법률전문가]:
개인정보보호법상 국외 이전 시 동의가 필요합니다.
약관 개정이 선행되어야 합니다.

🎤 [성능전문가]:
S3 자체는 좋지만 CDN 구성이 더 중요합니다.

[모더레이터]
투표 결과: 찬성 2 (긍정적 실무자, 성능전문가), 반대 2 (보안전문가, 법률전문가)

전문가 2명 이상 반대로 재논의가 필요합니다.
반대 의견의 핵심 쟁점을 먼저 해결하겠습니다.

**쟁점**: 국내 데이터 저장 요건 충족 방안
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[재논의 진행]

🎤 [긍정적 실무자]:
개인정보는 국내 전용 스토리지, 비개인정보는 글로벌 S3로 분리하면 어떨까요?

🎤 [보안전문가]:
분리 저장 + 국내 DR 구성이면 동의할 수 있습니다.

🎤 [법률전문가]:
데이터 분류 기준이 명확하면 법적 요건은 충족됩니다.

✅ CONCLUSION (재논의 후):
하이브리드 스토리지 합의
- 개인정보: 국내 전용 (NHN Cloud 등)
- 비개인정보: AWS S3 글로벌
- 조건: 데이터 분류 가이드라인 수립 필수
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```
