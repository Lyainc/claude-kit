# Unknown Discovery - Reference Guide

상세 절차, 질문 전략, 상태 관리, 트러블슈팅 가이드.

## 1. Phase 0: Context Analysis (상세)

### 1.1 대상 분석

분석 대상 유형별 접근:

| 유형 | 분석 포인트 | 초기 질문 방향 |
|------|------------|---------------|
| **프로젝트** | 목표, 범위, 제약조건 | 기술적 가정, 리소스 한계 |
| **기획안** | 핵심 가치, 타겟 사용자 | 시장 가정, 사용자 행동 예측 |
| **의사결정** | 선택지, 기준, 영향 범위 | 평가 기준의 완전성, 이해관계자 |
| **아이디어** | 핵심 컨셉, 차별점 | 실현 가능성, 수용성 |

### 1.2 도메인 추론 로직

```text
맥락 분석 → 키워드 감지 → 도메인 추론 → 사용자 확인

키워드 매핑:
- Tech: API, 시스템, 아키텍처, 성능, 서버, 데이터베이스, 코드
- Biz: 매출, 고객, 시장, 경쟁, ROI, 수익, 비용, 전략
- Creative: 컨셉, 디자인, 브랜드, 스토리, 콘텐츠, 크리에이티브
```

### 1.3 초기 질문 (AskUserQuestion)

```text
질문: "분석 대상의 도메인을 확인해주세요."

옵션:
1. Tech - 기술적 시스템, 아키텍처, 개발 관련
2. Biz - 비즈니스 전략, 시장, 수익 모델 관련
3. Creative - 창작물, 디자인, 콘텐츠 관련
4. Custom - 직접 중점 영역 지정
```

## 2. Phase 1: Interview Loop (상세)

### 2.1 질문 생성 전략

#### Core 4 영역별 질문 뱅크

**Assumptions (가정 검증)**:

```text
기본: "이것이 성립하려면 어떤 전제가 필요한가요?"
변형:
- "이 계획이 성공하려면 무엇이 반드시 참이어야 하나요?"
- "암묵적으로 가정하고 있는 것은 무엇인가요?"
- "이 전제가 틀리다면 어떻게 되나요?"
- "가장 위험한 가정은 무엇인가요?"

Why 체인:
- "왜 그 전제가 맞다고 생각하시나요?"
- "그 가정을 검증한 적이 있나요?"
- "반대 증거가 있다면 무엇일까요?"
```

**Trade-offs (트레이드오프)**:

```text
기본: "이 선택으로 포기하게 되는 것은 무엇인가요?"
변형:
- "이 방향을 선택하면 무엇을 희생하게 되나요?"
- "대안을 선택했다면 얻었을 것은?"
- "단기 vs 장기 관점에서 트레이드오프는?"
- "리소스(시간/비용/인력) 측면에서 희생되는 것은?"

Why 체인:
- "왜 그 희생이 감수할 만하다고 보시나요?"
- "그 가치를 포기해도 괜찮다고 판단한 근거는?"
```

**Edge Cases (극단 상황)**:

```text
기본: "10배 규모가 되면 어떤 문제가 생기나요?"
변형:
- "최악의 시나리오는 무엇인가요?"
- "가장 예상치 못한 사용 패턴은?"
- "시스템 한계에 도달하면?"
- "외부 환경이 급변하면?"

Why 체인:
- "왜 그 상황이 발생할 수 있다고 생각하시나요?"
- "그 상황에 대한 대비책은 있나요?"
```

**Blindspots (메타 인지)**:

```text
기본: "아직 질문하지 않은 것 중 중요한 것은 무엇인가요?"
변형:
- "논의하지 않았지만 마음에 걸리는 것은?"
- "다른 사람이라면 무엇을 물어볼까요?"
- "6개월 후 후회할 만한 누락은?"
- "직감적으로 불안한 부분은?"

Why 체인:
- "왜 그것이 중요하다고 느끼시나요?"
- "그 부분을 간과했던 이유가 있나요?"
```

#### Extended 4 영역별 질문 뱅크

**Feasibility (실현 가능성)**:

```text
- "가장 큰 기술적 장벽은 무엇인가요?"
- "필요한 리소스를 확보할 수 있나요?"
- "타임라인이 현실적인가요?"
- "필요한 역량이나 기술이 있나요?"
```

**Stakeholders (이해관계자)**:

```text
- "이 결정에 반대할 사람은 누구인가요?"
- "누구의 동의가 필수적인가요?"
- "영향을 받지만 의견이 반영되지 않은 사람은?"
- "숨겨진 이해관계자는 없나요?"
```

**Counterfactual (대안 탐색)**:

```text
- "완전히 다른 접근법이 있다면 무엇일까요?"
- "제약이 없다면 어떻게 하겠나요?"
- "경쟁자는 어떻게 접근할까요?"
- "5년 전/후라면 어떻게 달라질까요?"
```

**Dependencies (의존성)**:

```text
- "이것이 실패하면 어디가 영향받나요?"
- "어떤 외부 요소에 의존하고 있나요?"
- "변경 시 연쇄적으로 바뀌어야 하는 것은?"
- "가장 취약한 연결고리는?"
```

### 2.2 적응형 선택 로직

```text
불확실성 신호 감지:

1. Hedging Language (회피적 표현)
   - "아마", "글쎄", "잘 모르겠지만", "확실하진 않은데"
   - "probably", "maybe", "I'm not sure"
   → 해당 영역 후속 질문으로 심화

2. 짧은 응답
   - < 20자 (한글) / < 10 words (영어)
   → "좀 더 구체적으로 말씀해주시겠어요?"

3. 회피 반응
   - "나중에 생각해볼게", "별로 중요하지 않아"
   - "그건 괜찮아", "신경 안 써도 돼"
   → Why 체인 강화: "왜 중요하지 않다고 생각하시나요?"

4. 반복 응답
   - 이전 답변과 유사한 내용 반복
   → 다른 영역으로 전환 또는 포화 신호 카운트
```

### 2.3 체크포인트 프로세스

```text
매 4개 질문마다:

1. 진행 상황 요약
   "[영역 2/4 | 질문 8] 지금까지 다룬 내용:
   - 가정: [핵심 발견 1-2개]
   - 트레이드오프: [핵심 발견 1-2개]"

2. 계속 여부 확인
   "계속 진행할까요?"
   - 옵션 1: 계속 (다음 영역으로)
   - 옵션 2: 특정 영역 더 깊이
   - 옵션 3: 여기서 마무리
```

## 3. Phase 2: Synthesis (상세)

### 3.1 우선순위 분류 기준

| 우선순위 | 기준 | 예시 |
|---------|------|------|
| **Critical** | 프로젝트 실패 가능성, 즉시 조치 필요 | 법적 리스크, 핵심 가정 오류 |
| **Important** | 품질/성과에 영향, 계획 수정 필요 | 누락된 이해관계자, 리소스 부족 |
| **Nice-to-have** | 개선 기회, 선택적 대응 | 추가 기능 아이디어, 최적화 포인트 |

### 3.2 인사이트 추출 패턴

```text
발견된 Unknown Unknowns를 분석하여:

1. 패턴 인식
   - 여러 발견에서 공통 주제 추출
   - 연결된 이슈들 그룹화

2. 근본 원인 추론
   - 표면적 발견 → 근본 원인 추적
   - "왜 이것을 놓쳤는가?" 분석

3. 영향도 평가
   - 각 발견의 파급 효과 평가
   - 연쇄적 영향 매핑
```

## 4. Phase 3: Documentation (상세)

### 4.1 리포트 생성 규칙

```text
1. 발견 항목별 필수 요소:
   - 무엇을 발견했는가 (What)
   - 왜 중요한가 (Why)
   - 어떤 조치가 필요한가 (Action)

2. 우선순위별 정렬:
   - Critical → Important → Nice-to-have

3. 액션 아이템 구체화:
   - 담당자 제안 (가능한 경우)
   - 기한 제안 (가능한 경우)
   - 검증 방법 제안
```

### 4.2 메타데이터 기록

```text
- 총 질문 수
- 완료된 영역
- 포화 도달 여부
- 인터뷰 소요 시간 (체크포인트 수 × 4)
- 도메인 프리셋
- 주요 불확실성 신호 발생 지점
```

## 5. Troubleshooting

### 5.1 일반적인 문제 상황

| 상황 | 원인 | 해결책 |
|------|------|--------|
| 사용자가 질문에 답하지 않음 | 질문이 너무 추상적 | 구체적 예시와 함께 재질문 |
| 모든 답변이 짧음 | 대상에 대한 이해 부족 | Phase 0으로 돌아가 대상 재분석 |
| 반복적인 답변 | 해당 영역 탐색 완료 | 다른 영역으로 전환 |
| 사용자가 조기 종료 원함 | 시간 부족 또는 피로 | Soft Landing 후 부분 리포트 생성 |

### 5.2 도메인별 주의사항

**Tech 도메인**:

- 기술 용어 확인 필요 시 명확히 질문
- 시스템 경계와 외부 의존성 특히 주목
- 성능/보안 관련 가정 반드시 검증

**Biz 도메인**:

- 수치와 데이터 출처 확인
- 이해관계자 범위를 넓게 탐색
- 법적/규제 리스크 반드시 포함

**Creative 도메인**:

- 주관적 판단에 대한 근거 탐색
- 타겟 오디언스 가정 검증
- 트렌드/시장 변화 고려

## 6. Checklist

### 6.1 Phase 0 완료 체크리스트

- [ ] 분석 대상이 명확히 정의됨
- [ ] 도메인이 사용자와 확인됨
- [ ] 초기 맥락이 충분히 파악됨

### 6.2 Phase 1 완료 체크리스트

- [ ] Core 4 영역 모두 최소 1회 이상 탐색
- [ ] 각 질문에 Why 체인 수행
- [ ] 최소 2회 체크포인트 수행
- [ ] 포화 신호 또는 명시적 완료 확인
- [ ] 갭 체크 질문 수행

### 6.3 Phase 2 완료 체크리스트

- [ ] 모든 발견에 우선순위 태깅
- [ ] 핵심 인사이트 최소 2개 추출
- [ ] 패턴/연결고리 분석 완료

### 6.4 Phase 3 완료 체크리스트

- [ ] Discovery Report 생성
- [ ] Critical 항목에 액션 아이템 포함
- [ ] 메타데이터 기록 완료
